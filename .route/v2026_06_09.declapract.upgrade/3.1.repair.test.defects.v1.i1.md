### defect 5: Ref<typeof ...> type not generated for REFERENCE properties

**what:** type check failures in generated DAO files: `Ref<typeof Carriage>` expected but generator produced `Carriage` for reference properties in findByUnique method signatures.

**how:** the generator returned the raw domain object name for REFERENCE types. the upgrade changed domain-objects library expectations - REFERENCE properties to Domain Entities now require `Ref<typeof ...>` wrapper type.

```diff
# defineDaoFindByMethodCodeForDomainObject.ts getTypescriptTypeForDomainObjectProperty
- return referencedDomainObjectName;
+ return `Ref<typeof ${referencedDomainObjectName}>`;
```

**why:** the `getTypescriptTypeForDomainObjectProperty` function returned the raw domain object name for REFERENCE types. after the upgrade, the domain-objects library expects `Ref<typeof DomainEntity>` for typed references to Domain Entities.

**fix:** updated `defineDaoFindByMethodCodeForDomainObject.ts` to check if a REFERENCE property points to a Domain Entity and return `Ref<typeof ${name}>` instead of just the name.

### defect 6: Ref import not added when needed

**what:** TypeScript error: `Cannot find name 'Ref'` - the generator produced `Ref<typeof ...>` types but did not add `Ref` to the import statement.

**how:** imports were not updated to include Ref.

**fix:** added `hasDomainEntityReferenceInUniqueKey` check in `defineDaoFindByMethodCodeForDomainObject.ts` to detect when unique keys contain Domain Entity references, then conditionally add `Ref` to imports.

### defect 7: Ref<typeof ...> incorrectly applied to Domain Literals

**what:** type error: `Type 'typeof Price' does not satisfy constraint 'Refable'` - the initial fix applied `Ref<typeof ...>` to ALL REFERENCE types, but Domain Literals do not have `unique` keys required by `Refable`.

**how:** applied Ref to all reference types instead of only Domain Entities.

```diff
# defineDaoFindByMethodCodeForDomainObject.ts getTypescriptTypeForDomainObjectProperty
- if (domainObjectProperty.type === DomainObjectPropertyType.REFERENCE) {
-   return `Ref<typeof ${referencedDomainObjectName}>`;
- }
+ if (domainObjectProperty.type === DomainObjectPropertyType.REFERENCE) {
+   // only use Ref<typeof ...> for domain entities; domain literals use the type directly
+   if (
+     isDomainObjectReferenceProperty(domainObjectProperty) &&
+     domainObjectProperty.of.extends === DomainObjectVariant.DOMAIN_ENTITY
+   ) {
+     return `Ref<typeof ${referencedDomainObjectName}>`;
+   }
+   return referencedDomainObjectName;
+ }
```

**why:** Domain Literals (like `Price`, `Geocode`) do not have a `unique` key - they are identified by all their properties. the `Ref<typeof ...>` type requires the target to satisfy `Refable<any, any, any>` which requires a `unique` property. only Domain Entities have unique keys.

**fix:** added check for `domainObjectProperty.of.extends === DomainObjectVariant.DOMAIN_ENTITY` before `Ref<typeof ...>`. Domain Literals now use their raw type name directly.

### defect 10: pg DeprecationWarning treated as test error

**what:** integration test `should be able to apply the schema` failed because pg package writes DeprecationWarning to stderr.

**how:** test threw on any stderr output, but pg@8.14 writes `DeprecationWarning: Implicit coercion to Buffer is deprecated` to stderr.

```diff
# generate.integration.test.ts
- if (applyResult.stderr) throw new Error(applyResult.stderr);
+ const stderrWithoutWarnings = applyResult.stderr
+   .split('\n')
+   .filter((line) => !line.includes('DeprecationWarning'))
+   .filter((line) => !line.includes('--trace-deprecation'))
+   .join('\n')
+   .trim();
+ if (stderrWithoutWarnings) throw new Error(stderrWithoutWarnings);
```

**why:** Node.js deprecation warnings go to stderr but are not errors. the test conflated any stderr output with failure.

**fix:** filter out DeprecationWarning lines before check if stderr contains actual errors.

### defect 13: ESM/CJS interop type check failure

**what:** type check failed with TS1479: `The current file is a CommonJS module whose imports will produce 'require' calls; however, the referenced file is an ECMAScript module and cannot be imported with 'require'`.

**how:** exampleProject had `"type": "module"` to mark it ESM, but main project is CJS. TypeScript flagged the incompatibility.

```diff
# exampleProject/package.json
- "type": "module"
+ "type": "commonjs"
```

**why:** ESM and CJS have incompatible import semantics. CJS cannot `require()` ESM modules.

**fix:** changed exampleProject to `"type": "commonjs"` so both main project and test assets use same module system.

---

### defect 14: .ts extension imports not allowed in node16 module resolution

**what:** type check failed with TS5097: `An import path can only end with a '.ts' extension when 'allowImportingTsExtensions' is enabled`.

**how:** sql-schema-generator generates schema files with `.ts` extensions in imports (ESM pattern). TypeScript with `module: node16` rejects these unless `allowImportingTsExtensions` is enabled.

```diff
# tsconfig.json
"exclude": [
  ".agent/.cache",
  "dist",
  "coverage",
  "node_modules",
+ "src/domain.operations/__test_assets__/exampleProject/provision"
]
```

**why:** generated provision/schema files are runtime artifacts for sql-schema-control, not TypeScript source files for this project. they use ESM import syntax with .ts extensions which TypeScript in CJS mode rejects.

**fix:** excluded `exampleProject/provision` from TypeScript compilation.

### defect 15: depcheck reports unused dependencies

**what:** lint check failed with `Unused dependencies: simple-async-tasks` and `Unused devDependencies: procedure-fns`.

**how:** depcheck could not detect usage because:
- `simple-async-tasks` is only used in test assets domain objects
- `procedure-fns` is used in generated code templates (string literals)

```diff
# .depcheckrc.yml
+ - simple-async-tasks # used in test assets example project domain objects
+ - procedure-fns # used in generated code templates (generator outputs code that imports this)
```

**why:** depcheck analyzes static imports; it cannot detect usage in test assets or string literals for codegen templates.

**fix:** added both packages to .depcheckrc.yml ignores list with explanatory comments.

---

### defect 16: unit test fails to load generated ESM schema files

**what:** unit test `generate.test.ts` failed with `SyntaxError: Cannot use import statement outside a module` when sql-schema-generator tried to load generated declaration files.

**how:** generated files in `provision/schema/declarations/` use ESM imports (`import { x } from './y.ts'`) but exampleProject is CJS (defect 13 fix).

```json
# provision/schema/declarations/package.json
{
  "type": "module"
}
```

**why:** sql-schema-generator dynamically imports the generated declaration files at runtime. with the parent CJS context, Node.js tried to load ESM files as CJS and failed.

**fix:** added nested package.json in `provision/schema/declarations/` with `"type": "module"` to mark only that directory as ESM.
