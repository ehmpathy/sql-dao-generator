# vision: remove ts-node from sql-dao-generator

## the outcome world

### before

```
$ npx sql-dao-generator generate

TypeError: value.replace is not a function
    at normalizeSlashes (.../ts-node/dist/util.js:62:18)
    at Object.getExtendsConfigPath (.../ts-node/dist/ts-internals.js:24:54)
```

mechanic is blocked. CI is red. frustration builds. workaround docs pile up.

### after

```
$ npx sql-dao-generator generate

🔎 Loading domain objects: ...
🧠 Defining sql relationships: ...
🏗️  Generating the sql schema: ...
🔧 Generating schema-control config: ...
🔨 Generating data-access-objects: ...
🗜️  Generating typescript from sql: ...
```

works regardless of consumer's tsconfig setup. no compatibility matrix. no workarounds.

### the "aha" moment

> "wait, it just... works? even with my TypeScript 5.5 tsconfig with array extends?"

yes. because sql-dao-generator no longer reads the consumer's tsconfig at runtime. it runs from its own compiled dist/, completely isolated from consumer tsconfig quirks.

## groundwork

### external research

**node native typescript support:**
- node 22.6.0+ supports `--experimental-strip-types` (type stripping, no emit)
- node 23.6.0+ supports `--experimental-transform-types` (enum support)
- rhachet uses both flags: `--experimental-strip-types --experimental-transform-types`

source: rhachet bin/run.jit implementation (verified in repo)

### internal research

**current bin/run:**
```js
#!/usr/bin/env node
require('ts-node').register(); // <-- the problem
require('@oclif/core').run()...
```

ts-node.register() reads process.cwd()'s tsconfig.json, which may have array extends.

**rhachet bin/run.jit:**
```sh
#!/usr/bin/env -S node --experimental-strip-types --experimental-transform-types --disable-warning=ExperimentalWarning
require('../dist/contract/cli/invoke').invoke({ args: process.argv.slice(2) });
```

key insight: rhachet runs from dist/, not src/. no ts-node needed.

**ts-node in package.json:**
- listed in both dependencies AND devDependencies (line 95, line 142)
- removing from dependencies is the fix

**oclif config:**
```json
"oclif": {
  "commands": "./dist/contract/commands",  // <-- already points to dist/
  ...
}
```

the oclif config already expects dist/. the only reason ts-node exists is the bin/run entrypoint.

## what is awkward?

### the shebang limitation

node's typescript flags require `-S` for env to parse multiple args:
```sh
#!/usr/bin/env -S node --experimental-strip-types
```

this is posix-compliant but some ancient systems may not support `-S`.

**mitigated**: minimum node version is 12, all modern systems support this.

### contributors need to build first

local dev requires `npm run build` before testing cli changes.

**mitigated**:
- this is already the norm for oclif clis
- can add npm command: `npm run dev` that watches and rebuilds

### two entrypoints like rhachet?

rhachet has bin/run (shell dispatcher), bin/run.jit, bin/run.bun.

**for this fix**: keep it simple. one bin/run that runs from dist/. no bun support needed yet.
